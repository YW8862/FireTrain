# 智能消防技能训练评测系统：设计开发顺序文档（从编码到部署）

> 目标：把 `system.md` 的方案变成可执行落地路径。  
> 读者：项目负责人、开发同学、毕设实施者。  
> 使用方式：严格按阶段推进；每阶段都要有可验证产出，再进入下一阶段。

---

## 0. 文档使用说明

### 0.1 本文解决的问题

- 按什么顺序开发，能最快做出可演示版本？
- 每一步具体做什么、怎么做、做到什么算完成？
- 如何避免“功能都写了但联不起来、跑不起来”？

### 0.2 实施原则

- 先主链路后增强：优先打通“登录 -> 开始训练 -> 得分报告 -> 历史查询”。
- 先契约后实现：先定义 REST 接口与数据模型，再写业务代码。
- 先可跑后优化：先保证正确和稳定，再做性能与体验优化。
- 每阶段必须验收：没有验收结果不进入下一阶段。

### 0.3 本文交付结构

- 开发总顺序（宏观阶段）
- 每阶段详细步骤（输入/动作/产出/验收）
- 编码规范与任务拆分方式
- 测试、部署、上线后的运维顺序
- 常见风险与回退方案

## 1. 全流程开发总顺序（建议 12 周）

1. **阶段 A：工程初始化与基础设施**
2. **阶段 B：接口契约与数据模型**
3. **阶段 C：后端核心模块编码（先最小链路）**
4. **阶段 D：AI 推理与评分模块接入**
5. **阶段 E：前端页面与交互编码**
6. **阶段 F：前后端联调与端到端打通**
7. **阶段 G：测试、性能、稳定性整改**
8. **阶段 H：部署上线与演示封版**

> 推荐执行策略：每个阶段结束都出“里程碑产物”，避免返工。

#### 产出物

- `docs/环境配置.md`
- 前端与单体后端均可独立安装依赖

#### 验收标准

- 新机器按文档 30 分钟内可完成环境准备。

## 3. 阶段 B：接口契约与数据模型（第 1-2 周）

### B1. 先定数据库表与字段

#### 做什么

- 先把核心表定死，避免后续接口反复改。

#### 怎么做

- 落库表优先级：
  1. `users`
  2. `training_records`
  3. `action_logs`
  4. `training_statistics`
- 输出 SQL 初始化脚本和 ORM 模型。
- 明确关键约束：
  - 用户名、邮箱唯一
  - 外键关系正确
  - 时间字段统一时区策略（建议 UTC）

#### 产出物

- `docs/数据库设计.md`
- `db/init.sql` 或 migration 脚本

#### 验收标准

- 本地执行脚本可成功建表，且可完成一次插入/查询。

---

### B2. 先定 REST 接口契约与 Schema

#### 做什么

- 定义前后端 REST 接口，先契约再编码。

#### 怎么做

- 按业务域定义 REST 接口：
  - 用户接口
  - 训练接口
  - 统计接口
- 每个接口明确：
  - 请求路径与方法
  - 请求字段
  - 响应字段
  - 鉴权要求
  - 失败返回规范
- 用 `Pydantic` 建立请求/响应 Schema，作为代码与文档共同真源

#### 产出物

- `docs/API接口文档.md`
- `backend/app/schemas/` 下的请求/响应模型

#### 验收标准

- 至少 1 个 Demo REST 接口能成功调用并返回。

---

### B3. 后端路由与模块映射定义

#### 做什么

- 对前端暴露统一 REST 路径，并映射到后端内部模块。

#### 怎么做

- 按业务域分组：
  - `/api/user/*`
  - `/api/training/*`
  - `/api/stats/*`
- 每个 API 定义：
  - 入参校验规则（Pydantic）
  - 鉴权要求（是否需要 token）
  - 响应 JSON 结构
  - 错误码
- 形成接口文档（可用 OpenAPI 自动生成）。

#### 产出物

- `docs/API接口文档.md`
- 后端路由骨架代码

#### 验收标准

- 前端可依据接口文档完成 Mock 开发。

## 4. 阶段 C：后端核心模块编码（第 3-4 周）

> 顺序关键：**先应用骨架与公共能力 -> 用户模块 -> 训练模块 -> 统计模块**。

### C1. 用户模块（先做鉴权基础）

#### 做什么

- 实现注册、登录、用户信息查询/修改。

#### 怎么做

- 核心模块拆分：
  - `repository`（数据库访问）
  - `service`（业务逻辑）
  - `api`（REST Handler）
- 安全要求：
  - 密码 `hash` 存储（不可明文）
  - 登录成功返回 token（JWT）
- 重点异常：
  - 重复用户名/邮箱
  - 密码错误
  - 用户不存在

#### 产出物

- `backend/app/services/user_service.py`
- 对应单元测试

#### 验收标准

- REST 层面完成注册/登录/获取资料全链路可用。

---

### C2. 训练模块（训练主流程）

#### 做什么

- 实现训练任务创建、视频路径管理、结果保存与查询。

#### 怎么做

- 关键流程：
  1. 创建训练记录（状态 `created`）
  2. 接收上传后的视频元信息（或文件路径）
  3. 调用内部评分模块计算分数
  4. 回写总分、步骤分、反馈、状态 `done`
- 数据字段建议：
  - `training_id`
  - `user_id`
  - `video_path`
  - `status`
  - `total_score`
  - `step_scores`
  - `feedback`

#### 产出物

- `backend/app/services/training_service.py`
- 训练记录 CRUD 能力

#### 验收标准

- 单个训练任务从创建到结果落库可跑通（先用假评分）。

---

### C3. 单体应用入口与通用能力

#### 做什么

- 实现统一入口、鉴权中间件、错误码标准化和应用装配。

#### 怎么做

- 中间件：
  - 请求日志（含 trace_id）
  - JWT 解析
  - 统一异常处理
- 路由优先上线：
  - `POST /api/user/register`
  - `POST /api/user/login`
  - `POST /api/training/start`
  - `GET /api/training/result/{id}`
  - `GET /api/training/history`

#### 产出物

- `backend/app/main.py` 可启动并对外响应

#### 验收标准

- Postman 可在 REST 层完成最小业务链路验证。

---

### C4. 统计模块 ✅ (已完成)

#### 做什么

- 基于 `training_records` 生成个人统计、趋势、排行榜。

#### 怎么做

- 先实现 2 个必须接口：
  - 个人统计
  - 趋势数据
- 排行榜作为后续增强（可选）

#### 产出物

- `backend/app/services/analytics_service.py` 初版

#### 验收标准

- 有测试数据时，能返回可用于前端图表的数据结构。

## 5. 阶段 D：AI 推理与评分模块接入（第 4-5 周）

> 目标不是先追求最高精度，而是先可运行、可解释、可演示。

### D1. 评分模块骨架先行

#### 做什么

- 先返回“模拟评分”，保证主链路稳定。

#### 怎么做

- 实现内部评分服务函数：
  - 输入：视频路径/帧数据索引
  - 输出：总分、分项、反馈、动作日志
- 第一版用规则 + 固定阈值，不依赖训练模型。

#### 产出物

- 评分模块可在后端内部被调用

#### 验收标准

- 训练模块调用后能稳定落库评分结果。

---

### D2. MediaPipe 姿态分析接入

#### 做什么

- 按关键点计算姿态角度分，形成“姿态规范性”评分。

#### 怎么做

- 提取关键点：肩、肘、腕、髋。
- 实现角度函数（见系统文档附录思路）。
- 规则映射：
  - 偏差 <= 容差：满分
  - 超容差按档位扣分
- 输出细粒度指标，便于报告解释。

#### 产出物

- `pose_analyzer.py`
- 姿态评分单元测试

#### 验收标准

- 给定样例帧，角度结果和评分结果稳定可复现。

---

### D3. YOLOv8 检测接入（最小可行）

#### 做什么

- 实现灭火器目标检测，支撑动作识别判定。

#### 怎么做

- 第一版直接用预训练权重进行推理。
- 后续按 `8.5` 计划进行小规模微调。
- 推理优化：
  - 降分辨率（如 640）
  - 抽帧处理（如每秒 5-10 帧）

#### 产出物

- `detector.py`
- 基础检测日志（检出率、耗时）

#### 验收标准

- 在本地演示视频中可稳定检出灭火器主体。

---

### D4. 规则引擎与反馈生成

#### 做什么

- 实现动作完整性、姿态规范性、时效性三类评分合成。

#### 怎么做

- 评分流程：
  1. 动作序列识别
  2. 分项打分
  3. 总分与等级计算
  4. 反馈模板匹配
- 反馈必须可解释：
  - 指出“哪一步有问题”
  - 给出“如何改”

#### 产出物

- `rule_engine.py`
- `feedback_generator.py`

#### 验收标准

- 同一输入视频，多次运行分数偏差可控，反馈文本合理。

## 7. 阶段 F：前后端联调与端到端打通（第 7-8 周）

### F1. 联调顺序（必须按链路）

1. 用户注册/登录联调
2. 开始训练接口联调
3. 评分结果拉取联调
4. 历史记录联调
5. 统计页联调

### F2. 联调方法

- 使用统一测试账号与固定测试视频。
- 每次只改一个服务，避免多点问题叠加。
- 对每条 API 记录请求参数和响应样例。

### F3. 联调检查清单

- token 鉴权是否在所有受保护接口生效
- 错误码是否统一（前端能稳定识别）
- 评分结果字段是否完整（前端不做猜测）
- 空数据场景是否可显示（无历史记录时）

### F4. 产出与验收

- 产出：`docs/联调记录.md`
- 验收：从登录到查看报告，全流程可连续演示 3 次成功。

## 8. 阶段 G：测试、性能、稳定性整改（第 8-9 周）

### G1. 测试分层顺序

1. **单元测试**：规则计算、角度函数、鉴权逻辑
2. **接口测试**：REST 核心接口
3. **集成测试**：训练任务端到端
4. **回归测试**：关键功能重复验证

### G2. 核心测试用例建议

- 用户模块：
  - 重复注册、密码错误、token 过期
- 训练模块：
  - 视频缺失、评分超时、重复提交
- 评分模块：
  - 动作遗漏、顺序错误、姿态偏差、时间超限
- 统计模块：
  - 新用户无数据、历史较多时分页性能

### G3. 性能优化顺序

- 先定位瓶颈，再优化：
  1. 视频处理耗时
  2. 模型推理耗时
  3. 数据库查询慢 SQL
  4. 前端首屏加载
- 常见优化动作：
  - 抽帧 + 降分辨率
  - 模型预加载
  - 索引优化
  - 接口分页与缓存

### G4. 稳定性加固

- 增加超时与重试（区分读写）
- 关键日志落 trace_id
- 对外错误信息脱敏
- 增加服务健康检查接口 `/health`

### G5. 产出与验收

- 产出：
  - `docs/测试报告.md`
  - `docs/性能报告.md`
- 验收：
  - 单路视频平均处理延迟接近目标（<= 1s）
  - 关键路径无阻断性 bug

## 9. 阶段 H：部署上线与演示封版（第 9-12 周）

### H1. 部署方案选择

#### 方案 1：本地单机部署（毕设推荐）

- 适用：演示、验收、快速交付。
- 方式：Docker Compose 一键拉起所有组件。

#### 方案 2：前后端分离部署（进阶）

- 适用：有服务器资源、要更贴近生产。
- 方式：Nginx 对外，前端静态部署，单体后端独立进程部署。

---

### H2. 部署前准备

#### 做什么

- 固化版本、整理配置、准备初始化脚本。

#### 怎么做

- 版本冻结：
  - 前端包版本锁定
  - Python 依赖锁定
- 配置分层：
  - `dev` / `test` / `prod` 三套配置
- 数据准备：
  - 初始化 SQL
  - 默认管理员账号（可选）

#### 产出物

- `.env.prod`
- `deploy/` 脚本目录

#### 验收标准

- 在新机器能通过文档独立完成部署。

---

### H3. 容器化与编排

#### 做什么

- 为前端、单体后端、数据库编写 Docker 配置。

#### 怎么做

- 前端和后端分别提供 `Dockerfile`。
- Compose 编排组件：
  - `frontend`
  - `backend`
  - `mysql`（或 sqlite 挂载卷）
- 关键项：
  - 健康检查
  - 端口映射
  - 数据卷

#### 产出物

- `docker-compose.yml`
- 服务镜像构建脚本

#### 验收标准

- 一条命令启动后，前端可访问且可完成主链路。

### H4. 反向代理与 HTTPS（可选增强）

#### 做什么

- 用 Nginx 统一入口并代理 API。

#### 怎么做

- 路由规则：
  - `/` -> 前端静态资源
  - `/api/` -> 后端应用
- 生产建议：
  - 启用 HTTPS
  - 基础限流
  - 请求体大小限制（视频上传）

#### 产出物

- `deploy/nginx.conf`

#### 验收标准

- 前端与 API 在同一域名下可稳定访问。

## 13. 常见失败点与应对顺序

### 失败点 1：接口反复变更导致前后端互相等待

- 应对：接口先冻结最小版本，新增字段只追加不破坏兼容。

### 失败点 2：AI 精度不稳定拖垮进度

- 应对：先用规则与阈值保证可演示，模型优化并行推进。

### 失败点 3：部署阶段才发现环境不一致

- 应对：第 1 周就完成容器化最小版，后续持续更新。

### 失败点 4：没有日志，问题定位困难

- 应对：从第一天起统一日志格式 + trace_id。
