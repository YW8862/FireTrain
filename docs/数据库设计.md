# FireTrain 数据库设计

## 1. 设计目标

- 先冻结核心业务表，避免后续接口、统计和评分链路反复改表。
- 同时兼容当前开发默认的 SQLite 调试场景与后续 Docker 中的 MySQL 8.0 落地场景。
- 所有时间字段统一按 UTC 存储，接口输出时再转换为 ISO 8601 字符串。

## 2. 表关系总览

```text
users (1) ───────< training_records (N) ───────< action_logs (N)
  │
  └────────────── training_statistics (1)
```

关系说明：

- 一个用户可以有多条训练记录。
- 一条训练记录可以拆出多条动作识别日志。
- 一个用户仅保留一条聚合统计记录，便于首页看板和趋势统计直接查询。

## 4. 关键约束

### 4.1 唯一性

- `users.username` 唯一。
- `users.email` 唯一。
- `training_statistics.user_id` 唯一，保证用户统计表一对一。

### 4.2 外键关系

- `training_records.user_id -> users.id`
- `action_logs.record_id -> training_records.id`
- `training_statistics.user_id -> users.id`

所有外键统一设置 `ON DELETE CASCADE`，保证用户删除后其训练记录、动作日志和聚合统计不会残留脏数据。

### 4.3 时间策略

- SQL 初始化脚本执行时显式 `SET time_zone = '+00:00'`。
- 所有 `*_at` 或时间戳字段按 UTC 写入数据库。
- 后端 ORM 默认使用 UTC aware datetime。
- 前端展示时按浏览器或用户所在时区转换。

## 5. 索引设计

- `users`
  - 唯一索引：`username`
  - 唯一索引：`email`
- `training_records`
  - 组合索引：`(user_id, created_at)`，支持用户历史查询
  - 组合索引：`(status, created_at)`，支持后台任务筛选
- `action_logs`
  - 组合索引：`(record_id, step_index)`，支持按记录回放动作链路
  - 普通索引：`action_timestamp`

## 7. 示例插入与查询

```sql
INSERT INTO users (username, email, password_hash)
VALUES ('demo_user', 'demo@example.com', 'hashed-password');

INSERT INTO training_records (
    user_id,
    training_type,
    status,
    total_score,
    step_scores,
    duration_seconds
) VALUES (
    1,
    'extinguisher',
    'completed',
    88.50,
    JSON_OBJECT('check_pressure', 18.0, 'pull_pin', 20.0),
    35.20
);

SELECT tr.id, u.username, tr.total_score
FROM training_records AS tr
JOIN users AS u ON u.id = tr.user_id
WHERE tr.user_id = 1
ORDER BY tr.created_at DESC;
```
